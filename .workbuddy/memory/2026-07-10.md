### v0.4.25 — Docker 自动更新修復
- 用户反馈 `docker compose up -d` 仍拉取旧版本
- **根因**：`docker compose up -d` 不会自动拉取新镜像，若本地已有缓存则直接用旧的
- **修复**：docker-compose.yml workload-tool 服务加 `pull_policy: always`，每次 up 强制从 ghcr.io 拉最新
- **用户操作方式**：
  - 方案 A：`docker compose pull && docker compose up -d`（显式拉取）
  - 方案 B（推荐）：`docker compose up -d` 自动 pull（有 pull_policy: always）
  - 方案 C：Watchtower 每24h自动拉取重启（已有配置，无需手动）

### v0.4.25 — Docker 自动构建最终修复
- **根因**：`.github/workflows/docker-publish.yml` 一直在 `workload-tool-rust/v0.4.25/` 版本子目录里，GitHub Actions 要求在仓库根目录 `.github/workflows/` 才能触发。**从 v0.4.22 到 v0.4.25 的 tag 推送从未触发过 Docker 构建**
- **修复项**：
  1. 将 workflow 移到仓库根目录 `.github/workflows/docker-publish.yml`
  2. 镜像名从 `${{ github.repository }}`（展开为 `HotLL233/tj` → `ghcr.io/hotll233/tj`）改为 `hotll233/tj/workload-tool`
  3. build context 改为动态目录 `workload-tool-rust/${{ github.ref_name }}`
  4. `dockerfile:` 参数名修复为 `file:`（docker/build-push-action@v5 不认 dockerfile 参数）
  5. docker-compose.yml 镜像名同步改为 `ghcr.io/hotll233/tj/workload-tool:latest`
- **第 1 次构建失败**：`permission_denied: write_package` — GITHUB_TOKEN 只有仓库命名空间写权限，无法推送到 `ghcr.io/hotll233/workload-tool`（用户命名空间）
