### v0.3.24（2026-07-06 10:30）—— 修复导出/统计中实验室对应关系错误

#### 用户反馈
- 项目 A001 同时关联 410 和 411 两个实验室
- 用户在 **410 实验室** 选项卡下录入了 A001 的记录
- 统计页面显示正常（实验室列只显示 410）✅
- **导出 Excel 后**，A001 的数据同时出现在 410 和 411 下 ❌

#### 根因
1. `work_records` 表**没有 `group_id` 字段**，不存储录入时的实验室上下文
2. 导出/统计查询时，通过 `project_lab_links` 关联表获取实验室名称（返回项目关联的**所有**实验室）
3. 导致一条记录被"分配"到多个实验室，数量翻倍

#### 修复内容（方案A：保存实验室上下文）

**数据库迁移**（`src/db/migrations.rs`）：
- `ALTER TABLE work_records ADD COLUMN group_id INTEGER REFERENCES project_groups(id)`
- 旧记录的 `group_id` 设为 NULL（无法可靠推断）

**后端修改**（4个文件）：
1. `src/models/record.rs`：`RecordCreate` 新增 `group_id: Option<i64>`
2. `src/repo/record_repo.rs`：
   - `create()`：INSERT 时保存 `group_id`
   - `list()` + `get_by_id_on_conn()`：使用 `wr.group_id` 获取实验室名
3. `src/api/record_handler.rs`：`create()` 接受并传递 `group_id`
4. `src/api/export_data.rs` + `src/api/stats_handler.rs`：
   - 所有查询添加 `LEFT JOIN project_groups pg ON pg.id = wr.group_id`
   - `lab_name` 改为 `COALESCE(pg.name, '未知')`（优先使用 `wr.group_id`）

**前端修改**（2个文件）：
1. `project-root/frontend/src/api/client.ts`：`createRecord` API 新增 `group_id?: number`
2. `project-root/frontend/src/pages/EntryPage.tsx`：创建记录时传递当前 `groupId`

- 前端构建：`npm run build` ✅ 成功

#### 已知问题
1. **旧数据**（v0.3.24 之前创建的记录）：`group_id` 为 NULL，导出/统计时实验室列显示"未知"
   - 解决方案：重新录入这些记录（删除旧记录，在新版本中重新创建）
2. **新记录**（v0.3.24 之后）：`group_id` 正确保存，导出/统计只显示在对应实验室下

#### 关键经验
- **⚠️ 保存上下文到记录表**：当查询需要"记录创建时的状态"时，应将上下文保存到记录表（如 `group_id`），而不是通过关联表动态查询
- **⚠️ COALESCE 优先级**：`COALESCE(expr1, expr2, default)` 返回第一个非 NULL 值，适合实现"优先使用新字段，旧数据回退"的逻辑
- **⚠️ LEFT JOIN 保留 NULL**：使用 `LEFT JOIN project_groups` 时，如果 `wr.group_id` 为 NULL，`pg.name` 也为 NULL，`COALESCE` 会返回默认值
