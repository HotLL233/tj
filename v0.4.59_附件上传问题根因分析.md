# v0.4.59 附件上传问题根因分析报告

> 分析日期：2026-07-14 10:00
> 分析人：齐活林（Delivery Director）
> 对比范围：v0.4.27-A → v0.4.59（附件上传全部 7 个相关版本）

## TL;DR

**问题一直解决不了的根本原因：CHANGELOG 与实际代码严重脱节，修复方向从一开始就跑偏了。**

更具体地：
1. **v0.4.59 的 CHANGELOG 是失真的**——声称"用 useRef 修闭包陈旧"，但 `SampleInfoEntry.tsx` 在 v0.4.58 和 v0.4.59 **MD5 完全一致**（672e8f94…）。v0.4.59 实际只改了后端的 MIME 类型白名单，前端一行没动。
2. **前端代码里潜伏着一个真实 BUG**（删除按钮写错位置），但根本不是"只显示一个"的根因。
3. **用户报告的"只显示一个"很可能是操作/环境问题**，不是代码逻辑问题——但每次都被错误归因成"代码问题"，越改越乱。

## 3. 代码层面的真实 BUG（v0.4.58/59 共有）

`SampleInfoEntry.tsx` 第 411-415 行，删除单个文件按钮：

```tsx
<IconButton size="small" sx={{ p: 0.3 }} onClick={() => {
  const updated = [...files];
  updated.splice(fi, 1);
  updateRow(idx, '_pendingFiles', updated);  // ❌ 写到 rows._pendingFiles
}}><CloseIcon fontSize="small" /></IconButton>
```

**问题**：
- `updateRow` 修改的是 React `rows` state 的 `_pendingFiles` 字段
- 而 `files` 渲染时是从 `pendingFilesRef.current.get(idx)` 读
- 结果：**点删除按钮 → state 变了但 ref 没变 → 重新渲染时 ref 覆盖回去 → 删除无效**

**应该改成**：
```tsx
const arr = [...(pendingFilesRef.current.get(idx) || [])];
arr.splice(fi, 1);
pendingFilesRef.current.set(idx, arr);
setRows(prev => prev.map((r, i) => i === idx ? { ...r } : r));  // 仅触发重渲染
```

但这个 BUG **不会**导致"只显示一个"——它只让"删除单个文件"按钮无效。

## 4. "只显示一个" 的真实可能原因

我反复读了上传循环，没有发现能让 2 个文件变 1 个的代码逻辑。**最可能的真相是用户侧操作**：

### 4.1 浏览器对 `multiple` 的支持差异
- Chrome / Edge / Firefox：`<input type="file" multiple>` 通常按 Ctrl/Shift+点击 选多个，或直接框选多个
- 用户大概率**只点选了一个文件**，但主观认为"多选=应该多个"

### 4.2 上传请求被合并
- `for (const file of pendingFiles) { await uploadSampleInfoAttachment(...) }` 是顺序 await
- **如果用户点了一次"选择文件"，又点了一次"选择文件"——第二次 onChange 会叠加到 ref**（这部分代码是对的）
- 但**用户大概率只触发了一次 onChange**（因为多选 UI 一次就完事）

### 4.3 后端 MIME 拒绝
- v0.4.58 时期：后端只认 `application/pdf` / `application/msword` / `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- **Edge 浏览器对 .docx 的 MIME 常报 `application/octet-stream`** → **整批失败，前端 catch 掉但 snack 不一定显示**

这正是 v0.4.59 真正修的：**加了 `application/octet-stream` 和 `application/x-pdf` 到白名单**。但 CHANGELOG 把"加白名单"包装成了"修闭包"，导致修复被错误归因。

### 4.4 误判"只显示一个"的根因
- 提交后用 `attachmentsByRow[rec.id]?.length` 显示附件数
- 这个数是批量查的（`batchGetSampleInfoAttachments`），由 `useEffect([records])` 触发
- **timing 问题**：`load()` 拉 records 比 `upload` 完成后插入数据库早
- 第一次渲染时 `attachmentsByRow[rec.id]` 还是空 → 显示 0
- 用户看到 0 觉得"丢了"，但实际稍等就显示 1 个、2 个

## 5. 总结：为什么改不好

| 失败环节 | 表现 |
|---|---|
| ① CHANGELOG 与实际代码脱节 | v0.4.59 声称改了前端，实际没改；用户无法判断修复有没有真的生效 |
| ② 修复方向跑偏 | 用户报告"只显示一个"，但被反复当成"代码逻辑 bug"在改；实际可能是浏览器 MIME 拒绝（v0.4.59 改对了方向，但被错位包装） |
| ③ 缺乏回归测试 | 每次只改不测，结果"修了 3 个版本问题还在" |
| ④ 真实的删除按钮 BUG 被忽视 | 第 411-415 行写错位置，但从未被发现；用户每次都关注"上传数量"而没去测"删除单个" |
| ⑤ 显示端 timing 问题被忽略 | `attachmentsByRow` 是 effect 异步填充，提交后立刻看可能显示 0 → 用户误判"丢文件" |

## 6. 真正的修复清单（如果还要改）

1. **【必须】修正删除按钮 BUG**（第 411-415 行）—— 改写 `pendingFilesRef.current` 而不是 `updateRow`
2. **【建议】优化上传反馈**：每个文件上传完成都 setSnack 一次，让用户看到"1/2"、"2/2"
3. **【建议】优化附件数显示**：提交后立即 `setAttachmentsByRow(p => ({ ...p, [rec.id]: pendingFiles }))` 乐观更新，不等 batch 查
4. **【必须】CHANGELOG 必须写真改动**——v0.4.59 的 CHANGELOG 失真，违反工程纪律
5. **【建议】加单元测试**：模拟 `<input>` 两次 onChange，断言 `pendingFilesRef.current.get(0).length === 2`

## 7. 经验教训

- 改完后**必须用 `diff` 或 `md5sum` 验证**代码是否真改了，不能只看 CHANGELOG
- CHANGELOG 应当**自动从 git diff 摘要生成**，而不是手写
- 用户报告"现象"时，**先复现，再定位**——别在没复现的情况下反复改代码
- 真实的 BUG 往往不在用户报告的方向上（这次的删除按钮 BUG 才是真问题）
