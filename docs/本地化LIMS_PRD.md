## 0. 项目定位与增量边界

**一句话目标**：在用户已有的「检测核心」之上，增量新增 iLab 风格的「运营模块」，形成私有部署、离线可用、功能等效 iLabPlus 且**比 iLab 更完整（自带检测闭环）**的本地 LIMS。

### 0.1 复用 / 新增 / 补强 边界一览

| 类别 | 内容 | 说明 |
|------|------|------|
| ✅ **复用现有（不改动）** | 检测核心：`work_records`（分析检测）+ `rd_work_records`（研发送样） | 已存在，结构镜像、逻辑隔离 |
| ✅ **复用现有** | 主数据：`projects` / `methods` / `project_groups` | 新运营模块共用 |
| ✅ **复用现有** | Excel 导入（`/api/import/excel` + `import_mappings` 列头识别）/ 多 Sheet 导出预览 | 新模块库存/采购等直接复用导入导出范式 |
| ✅ **复用现有** | 客户端形态：系统托盘 + 单实例端口占用检测 + 自动备份（VACUUM INTO）+ Inno Setup | 安装包/启动方式不变 |
| ✅ **复用现有** | 双模块隔离范式（蓝=检测 / 橙=送样）与前端技术栈（React18+Vite5+MUI5+Tailwind3+BrowserRouter+axios） | 新增模块沿用「独立表+主数据+audit_log.module+handler/service/repo+独立卡片」范式 |
| 🆕 **新增（P0/P1/P2）** | RBAC、真实鉴权、仪器、库存、采购、审批流、经费、供应商、统计、文档 | 详见第 3 节需求池 |
| 🔧 **补强（跨模块）** | 审计完整性修复、业务状态机 | 覆盖现有检测核心 + 新运营模块，详见第 3 节补强项 |

### 0.2 现有架构关键事实（来自只读分析，PRD 据此设计增量）

- **后端**：Rust + Axum 0.7，路由 `api_router` 用 `.merge()` 平铺聚合在 `/api/...` 一级路径；统一响应 `ApiResponse{code,message,data}`（成功 code=0，错误 1001/2001/5000），**所有响应含错误均返回 HTTP 200**。
- **双模块隔离范式**：`work_records` 与 `rd_work_records` 结构镜像、完全独立表；共用主数据；审计用 `audit_log.module`（work='shared'/rd='rd'）逻辑隔离；新增业务域 = 新建独立表 + 共用主数据 + `audit_log.module` 隔离 + 一组 `xxx_handler/service/repo` + 前端独立卡片/入口（研发送样即克隆分析检测而来）。
- **数据层**：SQLite + r2d2 连接池 + WAL；每个写操作在 repo 层 `conn.transaction()` 显式事务，业务 SQL 与审计写入同事务。
- **三处已确认缺陷（PRD 必须覆盖）**：
  1. **鉴权是摆设**：`auth_handler.rs` 仅校验 `config.admin_user/admin_pass`（明文 `admin`/`admin123`，见 `config.rs`），登录返回 `tok_{user}_{hex}`；其余接口**不校验 token**，录入人靠前端 sessionStorage 自由填，无角色/权限隔离。
  2. **审计不完整不对称**：`audit_log` 表含 `module` 列。分析检测侧走 `execute_with_audit`（`audit.rs`）**不写 module → 默认 'shared'**；研发送样侧走 `log_on_conn_with_module(...,'rd')`。rd 审计视图过滤 `module IN ('rd','shared')` → **会泄露看到分析检测日志**。且审计无防篡改、可被软删（业务表 `deleted_at`）影响、无字段级 diff。
  3. **无业务状态机**：研发送样仅有 `status` 雏形（待取样/已取样）；检测侧无「任务→执行→复核→批准→报告」流程。

## 1. 产品目标与成功标准

**目标**：交付一个「检测+运营」一体的本地私有 LIMS —— 复用现有检测核心，增量补齐 iLabPlus 等效的运营闭环（库存/采购/仪器/审批/RBAC/真实鉴权），并修复鉴权与审计缺陷，使程序可私有部署、离线运行、可审计、有权限隔离。

**成功标准（首版 P0 上线即判定）**：

| # | 成功标准 | 度量 |
|---|----------|------|
| SC-1 | 所有写接口强制鉴权、按角色拦截 | 匿名/越权调用写接口返回 401/403，业务错误码统一 |
| SC-2 | 6 类角色权限隔离生效 | 实验员无法访问库管/采购/审批等越权接口（接口级拦截 100%） |
| SC-3 | 运营闭环跑通 | 申购→审批→订单→入库→库存增加→领用→出库 全链路可走通且有审计 |
| SC-4 | 仪器可预约/审批/保养 | 仪器台账+二维码+预约+审批+保养记录可落库 |
| SC-5 | 审计不可篡改且对称 | 任意写操作进入审计；rd 视图不泄露 work 日志；审计记录不可删除/伪造 |
| SC-6 | 离线/私有可用 | 无外网依赖（短信→站内信+本地 SMTP；云存储→本地磁盘/MinIO；微信→本地 PWA），启动即用 |

## 3. 需求池（P0 / P1 / P2 + 补强项）

> 优先级：P0=首版必做；P1=后做；P2=可选。每条含：编号 / 标题 / 描述 / 优先级 / 涉及模块 / 验收标准。

### 3.1 P0（首版必做）

#### P0-01 RBAC 权限体系 🆕
- **描述**：新增 `users` / `roles` / `role_permissions` 三表。预置 6 角色：系统管理员、主管、库管员、采购员、实验员、仪器管理员。每角色绑定接口级权限点（如 `inventory:write`、`purchase:approve`）。用户归属单一角色（首版不做多角色叠加）。
- **涉及模块**：鉴权/RBAC（全站）
- **验收标准**：
  1. 可在管理后台创建用户、分配 6 角色之一；
  2. `role_permissions` 定义的最小权限点可覆盖所有新增写接口；
  3. 越权调用被接口级中间件拦截（返回 403 + 业务码），不依赖前端隐藏。

#### P0-02 真实鉴权中间件 🔧（替代摆设鉴权）
- **描述**：引入 JWT 或 session 中间件，所有写接口（`.merge()` 聚合的 handler）统一校验登录态与角色；登录接口对接新 `users` 表（bcrypt/argon2 哈希口令），**废弃 `config.rs` 明文 `admin/admin123`**；前端登录页对接，token 存入安全存储（非 sessionStorage 明文自由填）。保留现有 `ApiResponse` 统一响应与 HTTP 200 约定。
- **涉及模块**：鉴权（全站）
- **验收标准**：
  1. 未登录/ token 失效调用写接口 → 401/403；
  2. 登录成功签发可校验 token，权限点随 token 或会话解析；
  3. 旧 `admin/admin123` 明文配置不再作为鉴权依据（首次启动强制改密或迁移）。

#### P0-03 仪器管理 🆕
- **描述**：新增 `instruments`（台账：名称/型号/位置/负责人/状态/照片路径/二维码串）+ `instrument_bookings`（预约：时间段/预约人/状态[待审批/通过/拒绝/完成]/审批人）+ 保养记录（可并入 bookings 或独立 `instrument_maintenances`）+ 使用记录。遵循双模块范式：`audit_log.module='instrument'`。
- **涉及模块**：仪器管理（运营·绿色系）
- **验收标准**：
  1. 仪器可增删改查、上传照片、生成可打印二维码；
  2. 预约可提交、审批人可审批、冲突时段拦截；
  3. 保养/使用记录可落库并进入审计。

#### P0-04 库存管理 🆕
- **描述**：新增 `inventory_items`（品名/品牌/单位/分类/标签/存放地/安全库存/近效期阈值）+ 批次表 `inventory_batches`（批号/数量/生产日期/有效期/单价）+ 领用/出库记录 `inventory_transactions`（类型：领用/出库/归还/报损）。支持分类/标签、低库存与近效期预警、单品全链路追溯、Excel 导出（复用现有导出范式）。
- **涉及模块**：库存管理（运营·绿色系）
- **验收标准**：
  1. 库存可手动/Excel 导入建立，支持多级分类与批量标签；
  2. 领用（可按批次）、统一出库、归还、报损流程可走通并写审计；
  3. 低库存/近效期触发预警（前端提示 + 站内信）；
  4. 单品从申购→入库→领用全链路可查询、可导出。

#### P0-05 采购管理 🆕
- **描述**：新增 `purchase_requisitions`（申购：申请人/物品/数量/用途/状态[待审批/通过/驳回]）+ `purchase_orders`（订单：供应商/明细/金额/状态[待审批/已批/已发送/已入库]）+ 入库动作（回写 `inventory_items`/`inventory_batches`）。审批由 P0-06 审批流驱动。移除在线商城/在线招商/短信通知（本地化替换见 iLab 报告 5.3）。
- **涉及模块**：采购管理（运营·绿色系）
- **验收标准**：
  1. 申购可提交、可并入/生成订单；
  2. 订单走审批（按金额阈值）→ 入库后库存自动增加；
  3. 订单/产品历史可查、可导出；
  4. 供应商通知采用站内信 + 本地 SMTP（不依赖外部短信网关）。

#### P0-06 审批流引擎 🆕
- **描述**：新增 `approval_rules`，支持三类独立规则（领用 / 申购 / 采购），规则维度 = 发起人 + 对象（分类/标签 或 金额阈值）+ 审批人（角色/具体人）。触发时生成待办，审批人在系统内（登录后）审批（替换 iLab 邮件异地审批）。规则匹配采用「最具体优先」策略。
- **涉及模块**：审批流（运营·绿色系，跨库存/采购）
- **验收标准**：
  1. 三类规则可配置且持久化；
  2. 领用/申购/采购提交后按规则生成审批待办；
  3. 审批人登录后可在「我的审批」处理，结果与通知（站内信）联动；
  4. 无匹配规则时默认直达（或按系统设置要求主管审批）。

### 3.4 补强项（跨模块，必须单列）🔧

#### P0-07 审计完整性修复（覆盖现有检测核心 + 新运营模块）
- **描述**：
  1. **修正 module 不对称**：统一 `audit_log.module` 语义 —— `work`=分析检测域、`rd`=研发送样域、`shared`=跨域主数据（projects/methods/groups）、新增运营域按模块命名（如 `instrument`/`inventory`/`purchase`/`approval`/`funds`/`supplier`）。分析检测侧改用 `log_with_module(...,'work')` 替代不写 module 的 `execute_with_audit`，杜绝 rd 视图泄露 work 日志。
  2. **不可篡改**：二选一（待确认）→ (a) WORM 表（append-only，`DELETE/UPDATE` 受限，仅追加修正记录）；或 (b) 哈希链（每条审计记录含前一条 hash，篡改即断链）。
  3. **字段级 diff**：审计写入 `before`/`after` JSON（变更字段 + 旧值/新值），至少覆盖核心写操作。
  4. **防软删绕过**：审计写入与业务同事务（现有已满足），且审计表自身不被业务 `deleted_at` 软删影响。
- **涉及模块**：审计（全站，含现有 work/rd）
- **验收标准**：
  1. work 审计视图（`module IN ('work','shared')`）不包含 rd 记录，反之亦然；
  2. 审计记录无法被删除/修改（WORM）或任何篡改可被哈希链检出；
  3. 核心写操作审计含字段级 before/after；
  4. 现有检测核心（work/rd）的写操作同样进入修正后的审计。

#### P0-08 业务状态机（覆盖检测/运营流程）
- **描述**：引入统一状态机 `task → 执行 → 复核 → 批准 → 报告`，应用于：(a) 现有检测核心补强（work_records 增加状态流转，rd 送样扩展状态）；(b) 新运营模块（申购/订单/预约的审批流转复用 P0-06 审批状态）。状态变更必须写审计且受角色权限约束（如仅复核人可「复核」）。
- **涉及模块**：状态机（跨检测核心 + 运营）
- **验收标准**：
  1. 检测记录可沿 任务→执行→复核→批准→报告 推进，越级操作被拦截；
  2. 状态变更进入审计（含操作人/时间）；
  3. 运营侧申购/订单/预约的审批状态与 P0-06 联动一致。

## 4. UI 设计稿（文字描述）

### 4.1 导航与主题组织
- **顶部/侧边导航**：沿用现有 BrowserRouter 单页结构。一级导航分区：
  - **检测核心（已有）**：蓝色系 —— 分析检测、研发送样（保持不变）。
  - **运营管理（新增）**：**绿色系** —— 仪表盘、库存、采购、仪器、审批、经费(P1)、供应商(P1)、统计(P1)、文档(P1)。
  - **系统（新增）**：RBAC 用户/角色、审批规则、审计日志、个人中心。
- **门户首页（Dashboard）**：卡片网格，延续现有「模块卡片」范式。检测区蓝卡、运营区绿卡；卡片显示待办数（如「待我审批 3」「近效期预警 2」）。

### 4.2 各新增模块页面结构（沿用「录入页 + 列表页 + 统计页」范式）
- **录入页**：表单 + 选择主数据（项目/方法/分类）+ 提交即触发审批（如需）；复用 MUI + Tailwind 现有组件库与蓝/橙以外的绿色主题变量。
- **列表页**：表格 + 筛选（状态/分类/时间段）+ 操作（查看/编辑/删除→软删写审计）+ 导出按钮（复用现有导出预览）。
- **统计页**：复用现有 `stats_handler` 范式，新增运营统计路由（绿色主题图表）。
- **审批中心**：「我的申请 / 我的审批」双栏，状态机可视化（步骤条展示 task→执行→复核→批准→报告 或 申购→审批→订单→入库）。
- **审计页**：按 `module` 过滤的只读时间线，展示字段级 diff（补强后）；rd 视图严格不含 work 记录。
