## 二、已修复问题清单（逐条对照 v0.4.47 审查报告 17 条改进建议）

### P0 — 严重安全问题

#### 1. ✅ 部分修复：JWT_SECRET 硬编码

- **现状**：`auth_service.rs` 第 10-13 行，从 `const JWT_SECRET: &str` 改为函数 `fn jwt_secret() -> String`，优先读取环境变量 `JWT_SECRET`，未设置时 fallback 到硬编码默认值 `"workload-tool-jwt-secret-v0.4.29"`。
- **评价**：有改进，支持环境变量覆盖，但内置 fallback 仍是硬编码。生产环境如忘记设置 `JWT_SECRET` 环境变量，所有实例仍共享同一默认密钥。

#### 2. ✅ 已修复：audit_repo.rs SQL 注入漏洞

- **现状**：`list` 函数中，`module` 参数使用白名单硬编码字符串（`"work"`、`"rd"`、`"shared"`），无效值直接返回 `AppError::Validation` 错误。不再有 `format!` 拼接用户输入到 SQL 中。
- **评价**：彻底修复。白名单 + 参数化查询，无绕过风险。

#### 3. ❌ 未修复（部分缓解）：默认管理员密码 admin123

- **现状**：`config.rs` 中 `default_admin_pass()` 仍返回 `"admin123"`。v0.4.52 新增了 `ADMIN_PASSWORD` 环境变量支持（Docker/Linux 部署时可覆盖），且 config.toml 也可以覆盖该值。但未强制首次启动修改密码。
- **评价**：有缓解手段，但本地部署用户若未配置 config.toml 且未设置环境变量，仍然使用 admin/admin123 弱凭据。

### P1 — 架构缺陷

#### 4. ❌ 未修复：全局 JWT 鉴权 middleware

- **现状**：`api/mod.rs` 中无任何 middleware 层。各 handler 各自调用 `extract_claims_from_headers` 或 `extract_admin_from_headers` 做鉴权。`settings_handler.rs` 的 `list` 和 `get_by_key` 端点完全没有鉴权保护，任何未登录用户可查看所有系统设置。
- **评价**：与 v0.4.47 完全一致，未做任何改动。

#### 5. ❌ 未修复：ManagePage.tsx 未拆分

- **现状**：`ManagePage.tsx` 从 v0.4.47 的 158KB 增至 v0.4.52 的 159KB（略增），仍是一个文件承载项目管理、方法管理、分组管理、用户管理、角色管理等多个独立功能。
- **评价**：拒不拆分，反而变大。`StatsPage.tsx`（76KB）和 `SampleStatsPage.tsx`（72KB）同样未拆。

#### 6. ✅ 已修复：清理 frontend/src/src/ 嵌套目录

- **现状**：v0.4.52 的 `frontend/src/` 下已无 `src/src/` 嵌套目录。
- **评价**：彻底清理，源码仅保留在 `frontend/src/` 一层。

### P2 — 代码质量改进

#### 7. ❌ 未修复：无单元测试

- **现状**：Rust 源码和前端代码中均无任何测试文件（`#[cfg(test)]`、`_test.rs`、`*.test.ts`、`*.spec.ts` 均不存在）。
- **评价**：未做任何测试覆盖工作。

#### 8. ❌ 未修复：export_write.rs 重复模板代码

- **现状**：`export_write.rs` 仍为 974 行，11 个 Sheet 的写入逻辑仍是重复模板代码，未使用策略模式或 Builder 模式重构。
- **评价**：代码与 v0.4.47 完全一致。

#### 9. ❌ 未修复：backup_handler.rs 错误响应

- **现状**：`backup_handler.rs` 中多处仍使用 `ApiResponse::<String>::ok_msg(format!("{}", e))` 返回错误——即业务错误通过 HTTP 200 + code=0 返回，前端无法区分成功和失败。鉴权失败也走此路径。
- **评价**：与 v0.4.47 完全一致。

#### 10. ❌ 未修复：CORS 仍是 permissive()

- **现状**：`main.rs` 中 `.layer(CorsLayer::permissive())` 未做任何修改。
- **评价**：生产环境仍存在 CSRF 风险。

#### 11. ❌ 未修复：重复鉴权代码未提取

- **现状**：`extract_claims_from_headers` 和 `extract_admin_from_headers` 在多个 handler 文件中重复定义，未提取为公共函数或 middleware。
- **评价**：与 v0.4.47 完全一致。

#### 12. ❌ 未修复：未强制首次修改默认密码

- **现状**：未添加首次启动修改密码的强制逻辑。种子数据仍直接插入 admin 账号。
- **评价**：与 v0.4.47 完全一致。

## 五、新发现的问题

### 5.1 后端新问题

#### 🟡 所有错误类型仍返回 HTTP 200

- **位置**：`error.rs` 第 48 行 `(StatusCode::OK, Json(body)).into_response()`
- **问题**：`AppError::IntoResponse` 实现中，所有错误（包括 `Forbidden`、`Validation`、`Internal`）都使用 `StatusCode::OK`（200）返回，仅在 JSON body 的 `code` 字段区分。前端无法通过 HTTP 状态码判断成功/失败，所有响应都是 200。
- **风险**：浏览器 DevTools Network 面板无法直观看到错误请求，中间件/反向代理无法根据 HTTP 状态码做限流或告警。

#### 🟡 已知 UTF-8 截断 panic 风险未根本解决

- **位置**：`settings_handler.rs` 第 119-124 行（注释原文 `"在字符边界安全截断（避免 UTF-8 多字节字符被劈开导致 panic）"`）
- **问题**：该注释在 v0.4.47 审查报告中已被指出，v0.4.52 中仍然存在，说明该 panic 风险代码仍在使用。

#### 🟢 Dockerfile 环境变量覆盖更完善

- `config.rs` 的 `load()` 方法被重构为增量式加载：先加载默认值 → 环境变量 `ADMIN_PASSWORD` → config.toml 逐字段覆盖。相比 v0.4.47 粗暴的 `toml::from_str(&c).unwrap_or_default()` 更为合理。

### 5.2 前端新问题

#### 🟡 ManagePage 不减反增

- `ManagePage.tsx` 从 158KB 增至 159KB，说明新功能仍然直接塞入这个大文件，完全没有拆分的意图。

#### 🟡 新增前端组件增加了代码耦合

- 新增的 `ManageExportConfig.tsx` 和 `ManageFormConfig.tsx` 是对 ManagePage 的补充组件，但它们没有独立路由，仍然依赖 ManagePage 承载。进一步加深了 ManagePage 的单体架构耦合。

#### 🟢 前端功能增强

- 新增 tailwindcss 和 @dnd-kit 拖拽库，为后续 UI 改进做了铺垫。
- `UserContext.tsx` 增加了 `remember me` 功能，支持 localStorage/sessionStorage 双模式存储。
- 新增了 `PermissionGate.tsx` 权限门控组件，可用声明式方式控制 UI 元素的可见性。

## 六、综合评估

### 6.1 版本差异小结

v0.4.47 → v0.4.52 是一次**小幅度安全修补 + 功能增补**版本迭代（共迭代 5 个版本号）。变更范围极小——87 个 Rust 源文件中仅 5 个有实质性改动，42 个前端 TSX 文件中仅新增 2 个。

**核心改动：**
- 安全层面：修复了 SQL 注入（P0，彻底），部分缓解了 JWT 密钥硬编码和默认密码问题
- 功能层面：导出 Excel 支持从 system_settings 加载模板配置（Sheet 可见性/颜色/名称）
- 数据库层面：新增 v0.4.49 迁移，添加表单字段定义配置
- 整洁层面：清理了嵌套目录和 .bak 文件

### 6.2 总体评价

| 维度 | 评分（满分 10） | 变化 |
|------|---------------|------|
| 安全性 | 5/10 → **6/10** | +1（修复 SQL 注入，部分缓解 JWT/密码问题） |
| 可维护性 | 5/10 → **5/10** | 无变化（大组件未拆、重复代码未重构） |
| 代码整洁 | 6/10 → **7/10** | +1（清理嵌套目录和 .bak 文件） |
| 测试覆盖 | 1/10 → **1/10** | 无变化（仍无任何测试） |
| 架构设计 | 6/10 → **6/10** | 无变化 |

**综合评价：中等偏下。** 本次迭代修复了最严重的 SQL 注入漏洞，部分缓解了 JWT 密钥硬编码问题，是值得肯定的安全改进。但 17 条改进建议中仅彻底修复了 2 条（SQL 注入、嵌套目录），其余 15 条均未有效解决。特别是 3 条 P0 中仍有 1 条（默认弱密码）未彻底修复，3 条 P1 中仍有 2 条（全局鉴权 middleware、前端大组件拆分）未触及。代码规模、架构稳定性、测试覆盖等核心质量指标均原地踏步，建议在后续版本中优先补齐这些短板。

### 6.3 关键建议（按优先级）

| 优先级 | 建议 | 领域 |
|--------|------|------|
| **P0** | 从代码中移除 JWT_SECRET 硬编码 fallback，启动时未设置环境变量则拒绝启动 | 安全 |
| **P0** | 默认密码改为随机生成并首次启动时打印到控制台，或强制要求首次修改 | 安全 |
| **P1** | 添加全局 JWT 鉴权 Axum middleware（`/api/auth/login` 免鉴权） | 安全/架构 |
| **P1** | 为所有 API（特别是 settings_handler 的 `list`/`get_by_key`）补全鉴权 | 安全 |
| **P1** | 拆分 ManagePage.tsx 为多个独立路由页面 | 可维护性 |
| **P1** | 添加 Rust 集成测试覆盖核心 API 路径 | 质量保证 |
| **P2** | CORS 改为白名单域名而非 permissive() | 安全 |
| **P2** | 修复 backup_handler 的错误响应格式，使用正确 HTTP 状态码 | 一致性 |
| **P2** | 修复 error.rs 中所有错误类型返回 HTTP 200 的问题 | 正确性 |
