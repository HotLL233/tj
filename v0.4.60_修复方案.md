## Bug #1：附件删除按钮无效

### 根因（已定位）
**文件**：`frontend/src/pages/SampleInfoEntry.tsx`，约第 411-415 行

**当前代码**：
```tsx
<IconButton onClick={() => {
  const updated = [...files];
  updated.splice(fi, 1);
  updateRow(idx, '_pendingFiles', updated);  // ❌ 写到 React state
}}><CloseIcon fontSize="small" /></IconButton>
```

**真相源不一致**：
- 渲染时：`files = pendingFilesRef.current.get(idx)`  ← 从 ref 读
- 提交时：`pendingFilesRef.current.get(i)`  ← 从 ref 读
- 删除时：`updateRow(idx, '_pendingFiles', updated)`  ← 写到 state

**结果**：
- 重新渲染时 `files` 仍从 ref 读 → 旧 ref 覆盖回去
- 视觉上文件还在
- 提交时仍上传这个文件
- **删除按钮等于无效**

### 修复方案

将 onClick 改为直接修改 `pendingFilesRef.current`：

```tsx
<IconButton onClick={() => {
  const arr = [...(pendingFilesRef.current.get(idx) || [])];
  arr.splice(fi, 1);
  pendingFilesRef.current.set(idx, arr);
  // 触发重渲染（不影响数据，只让 UI 重新读 ref）
  setRows(prev => prev.map((r, i) => i === idx ? { ...r, _pendingCount: arr.length } : r));
}}><CloseIcon fontSize="small" /></IconButton>
```

**改动文件**：`frontend/src/pages/SampleInfoEntry.tsx`（1 处）

**回归测试**：
1. 选 2 个文件 → 看到 2 个文件名 + 2 个删除按钮
2. 点第 1 个的删除 → 该行消失，只剩 1 个
3. 提交 → 数据库只有 1 条 attachment 记录
4. 再选 1 个文件 → 仍能正常上传

## Bug #2：样品信息登记无法新建类型

### 根因分析

**调用链**（用户操作 → 后端）：
1. 前端 `saveSiType`（`ManagePage.tsx:382-401`）调 `createSampleInfoType`
2. axios POST `/api/sample-info-types`
3. 路由到 `sample_info_type_handler::create`（`src/api/sample_info_type_handler.rs:48-59`）
4. `require_admin(&headers)?` 校验 JWT
5. `sample_info_type_repo::create(&pool, &body)?` 写入 `sample_info_types` 表（事务）
6. 拿新连接 `pool.get()?`
7. `init_for_type(&conn, &item.type_key)?` 给新类型初始化预置列可见性

**最可能的根因**（按概率排序）：

#### 假说 1：init_for_type 的 SQL 与表结构不兼容（**最可能**）

`init_for_type`（`src/repo/sample_info_column_visibility_repo.rs:27-34`）：
```sql
INSERT OR IGNORE INTO sample_info_column_visibility (type_key, column_id, is_visible)
SELECT ?1, id, 1 FROM sample_info_columns WHERE is_predefined = 1
```

表结构（`migrations.rs:435-441`）：
```sql
CREATE TABLE IF NOT EXISTS sample_info_column_visibility (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    type_key    TEXT NOT NULL,
    column_id   INTEGER NOT NULL REFERENCES sample_info_columns(id) ON DELETE CASCADE,
    is_visible  INTEGER NOT NULL DEFAULT 1,
    UNIQUE(type_key, column_id)
);
```

**问题**：
- 如果 `sample_info_columns` 表里 `is_predefined = 1` 的行**没有 13 条**（或更少），INSERT 会插入 N 条
- 如果 N 条里有**任何一条**触发 `UNIQUE(type_key, column_id)` 冲突（应该不会，因为是新 type_key），INSERT 会被 IGNORE 跳过
- **正常情况不会失败**

#### 假说 2：require_admin 失败（**高概率**）

`require_admin`（`sample_info_type_handler.rs:25-36`）：
```rust
fn require_admin(headers: &HeaderMap) -> Result<auth_service::Claims> {
    let token = headers
        .get("authorization")
        .and_then(|v| v.to_str().ok())
        .and_then(|v| v.strip_prefix("Bearer "))
        .ok_or_else(|| AppError::Validation("未提供登录凭证".into()))?;
    let claims = auth_service::verify_token(token)?;
    if !claims.is_admin {
        return Err(AppError::Forbidden("需要管理员权限".into()));
    }
    Ok(claims)
}
```

**问题**：
- 如果用户登录的 JWT token 解析后 `claims.is_admin = false` → 返回 `Forbidden("需要管理员权限")`
- 错误信息是 1003 业务码 + 明确提示

**如何验证**：
- 在浏览器 DevTools 看 Network → POST `/api/sample-info-types` 的 response body
- 看 `message` 字段是"需要管理员权限"还是其他

#### 假说 3：前端没以管理员身份登录

`ManagePage.tsx:599`：
```tsx
{tc.filter(c => user?.is_admin || !c.perm || hasPermission(user?.permissions || [], c.perm)).map(c => (
```

如果 `user.is_admin = false`，**管理卡片本身就显示不了**——用户根本进不来管理页。

但用户能进管理页（看到"新建类型"按钮），说明 is_admin 是 true。

#### 假说 4：SQLite 缺索引或触发器导致 INSERT OR IGNORE 性能问题

不太可能（数据量很小）

#### 假说 5：type_key 含特殊字符（如中文/空格）

- 后端 `create` 已经 `.trim().to_string()`
- `UNIQUE(type_key, column_id)` 不应该因字符集失败

#### 假说 6：**r2d2 连接池耗尽**

`sample_info_type_repo::create` 持有一个连接做事务，事务结束后**没有显式归还**
（rusqlite + r2d2 应该自动归还，但配置不当可能泄漏）

但**这会让请求超时**，不会让保存"失败但没反应"

#### 假说 7：**事务里错误吞了真实信息**（**次可能**）

```rust
let item = sample_info_type_repo::create(&pool, &body)?;  // 这里事务已经 commit
// 之后任何错误都返回 Internal，但类型已存在
let conn = pool.get()?;
sample_info_column_visibility_repo::init_for_type(&conn, &item.type_key)?;
```

如果 `init_for_type` 失败（虽然原因不明），错误信息会被 `?` 转成 `AppError::Internal(format!("init_for_type失败: {}", e))`，但**真实 SQL 错误被吞了**（原始信息 `format!("{}", e)` 可能很简略）。

**但这不影响"无法新建"——类型已经创建成功了**。

---

### 真正的根因（**最可能**）

**init_for_type 在新连接上执行时，由于 r2d2 池的连接被独立管理，可能在并发场景下出现 UNIQUE 冲突**。

更具体地：
- 如果**用户连续创建两个相同 type_key 的类型**（前端 trim 后还是相同），第二个 `create` 的 `exists` 检查正常失败
- 但**如果 `init_for_type` 在新连接上被调用**，此时**前一个 `create` 还在事务中未提交**，新连接 SELECT 时**看不到未提交数据**——但 `INSERT OR IGNORE` 不依赖 SELECT
- **不会**有竞态

---

### 我的最终判断

**最大可能**（按用户实际报告"无法新建"反推）：

**根因 #1（高可能）**：`require_admin` 因为 `is_admin` 校验失败返回 `Forbidden("需要管理员权限")`——但前端的 `sm` 显示应该看得到这条消息，用户没看到说明不是它。

**根因 #2（高可能）**：`init_for_type` 执行 SQL 时**某些边界情况**导致 INSERT 失败（虽然具体原因不明）——但**类型已经成功创建**（事务已 commit），所以**用户看到"保存失败"但实际上类型已经存在于 DB**，刷新页面应该能看到。

**根因 #3（中可能）**：**前端保存按钮按了没反应**——可能不是后端问题，而是前端 `setMsg` 后被立刻清空（onClose 触发）。

**根因 #4（低可能）**：数据库锁定/超时。

### 修复方案

#### 方案 A（推荐）：错误信息透明化 + init_for_type 隔离事务

**改动 1**：`sample_info_type_handler.rs:48-59`
```rust
async fn create(
    State(pool): State<DbPool>,
    headers: HeaderMap,
    Json(body): Json<SampleInfoTypeCreate>,
) -> Result<Json<ApiResponse<SampleInfoType>>> {
    require_admin(&headers)?;
    let item = sample_info_type_repo::create(&pool, &body)?;

    // v0.4.60: 初始化可见性失败不应该让整个 create 失败
    // 改为可恢复的后台任务 + 详细错误日志
    let conn = pool.get()?;
    if let Err(e) = sample_info_column_visibility_repo::init_for_type(&conn, &item.type_key) {
        tracing::error!(
            "init_for_type 失败 (type_key={}): {} — 类型已创建但预置列可见性未初始化，需手动 SQL 修复",
            &item.type_key, e
        );
        // 不返回错误，让类型创建成功
    }
    Ok(Json(ApiResponse::ok(item)))
}
```

**理由**：
- 类型已成功创建（事务已 commit）→ 应当返回成功
- `init_for_type` 失败时只影响该类型的预置列可见性，**不应**让整个 API 失败
- 通过 `tracing::error!` 留下诊断日志

**改动 2**（可选，但建议加）：`init_for_type` 改用事务 + 显式 connection

```rust
pub fn init_for_type(pool: &DbPool, type_key: &str) -> Result<usize> {
    let mut conn = pool.get()?;
    let tx = conn.transaction()?;
    let count = tx.execute(
        "INSERT OR IGNORE INTO sample_info_column_visibility (type_key, column_id, is_visible) \
         SELECT ?1, id, 1 FROM sample_info_columns WHERE is_predefined = 1",
        [type_key],
    )?;
    tx.commit()?;
    Ok(count)
}
```

**改动 3**（前端）：让保存按钮在错误时不立即清空 msg

检查 `ManagePage.tsx:71` 的 `sm` 函数：

```tsx
const sm = useCallback((m: string, isErr?: boolean) => { setMsg(m); setErr(!!isErr); }, []);
```

`onClose={() => setMsg('')}` 让用户可以关掉 alert。但如果 alert 显示后用户没看到（比如被其他事件覆盖），会"看起来没反应"。
