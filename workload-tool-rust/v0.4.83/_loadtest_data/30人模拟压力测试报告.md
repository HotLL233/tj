# 30 人模拟使用压力测试报告

## 测试范围
- 模拟总人数：30 人。
- 分析检测人员：10 人。
- 研发送样人员：10 人。
- 其他部门送样人员：10 人。
- 取样人员：5 人。

## 模拟数据
- 分析检测工作量记录：1500 条，写入 `work_records`。
- 研发送样工作量记录：1800 条，写入 `rd_work_records`。
- 其他样品信息登记记录：900 条，写入 `sample_info_records`。
- 总业务记录：4200 条。

## 测试方法
- 使用隔离数据库 `_loadtest_data/workload_loadtest.db`，未使用正式数据库。
- 按软件当前表结构和统计逻辑模拟写入、取样状态、统计查询。
- 覆盖查询：
  - 分析检测总览和按人员统计。
  - 研发送样总览、按人员、按事业部统计。
  - 样品信息登记按状态统计。
  - 研发送样待取样/已取样统计。
  - 取样人员工作量统计。

## 性能结果
- 4200 条记录生成总耗时：约 385 ms。
- 120 轮统计查询，16 并发：0 错误。
  - 平均耗时：13.87 ms。
  - P95：21.70 ms。
  - 最大耗时：27.12 ms。
- 1000 轮统计查询，32 并发：0 错误。
  - 平均耗时：21.20 ms。
  - P95：33.29 ms。
  - 最大耗时：44.54 ms。

## 数据隔离检查
- 分析检测统计只查询 `work_records`。
- 研发送样统计只查询 `rd_work_records`。
- 样品信息登记统计只查询 `sample_info_records`。
- 三类数据在模拟中互不混用。

## 结果判断
- 以 30 人、4200 条业务记录、32 并发统计查询的规模看，当前 SQLite + 本地单机架构可以承受。
- 统计查询响应时间处于可接受范围，未出现查询错误。
- 研发送样和分析检测的数据隔离逻辑在数据库层验证正常。

## 当前能力边界
- 当前打包程序的 `/api/auth/login` 只接受配置中的 admin 账号登录，普通用户不能通过该入口获得 token。
- 因此本次未做真实 30 个账号分别登录后的 API 压力测试。
- 研发送样取样接口当前取样人来自登录 token；在真实 API 场景中，如果只能 admin 登录，取样人会记录为 admin。数据库层已模拟了不同取样人分布。

## 建议
- 若要做真实多用户登录压测，应先修复登录入口，让普通用户账号可正常登录并获得 token。
- 建议后续增加一个正式压测脚本：自动创建用户、登录、录入、取样、统计、导出，并输出接口级错误率和耗时。
