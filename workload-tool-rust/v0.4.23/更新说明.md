# 本地化LIMS v0.4.23 更新说明

> 版本：0.4.23（基于 `本地化LIMS/v0.4.22-B` 独立副本，版本隔离于 `workload-tool-rust/v0.4.23`）

## 本版要点
- **用户分级与入口自定义**：管理员在后台为用户分配角色（`admin` / `analyst` / `experimenter` 等），并为每个角色勾选其可见的导航入口；侧边栏按角色动态渲染，未授权入口不显示。权限点集中在 `frontend/src/constants/permissions.ts`，角色-权限映射在后台 `AdminRolesPage` 可编辑。
- 版本号三处一致：`backend/Cargo.toml`、`backend/build_installer.iss`、`frontend/package.json` 均升至 `0.4.23`。
- 安装包沿用固定 `AppId`，支持覆盖安装上一版本；数据库（SQLite）保留不删。
> 语言：中文
> 定位：路线 C 混合本地 LIMS（检测核心 + iLab 风格运营模块），纯本地、离线、私有部署。

## 一、本版变更清单

### 1. 修复后端编译错误（F1–F4，共 9 处编译错误 → 零 error）
基于《本地化LIMS_可行方案.md》§3.2 的修复清单，逐条落地：
- **F1**：`api/instrument_handler.rs`、`api/supplier_handler.rs` 中 `use ...routing::delete` 与同名 `async fn delete` 冲突 → 导入改名 `delete as delete_route`（路由处改用 `delete_route` 方法，handler 函数名 `delete` 保持不变）。
- **F2**：`db/seed.rs` / `service/auth_service.rs` 的 `OsRng` 受 `rand_core` 的 `getrandom` feature 门控 → `Cargo.toml` 给 `rand_core` 开启 `features=["getrandom"]`；`seed.rs` 导入改为 `use rand_core::OsRng;`（与 `auth_service.rs` 一致）。
- **F3**：`middleware/auth.rs` 的 `AuthedUser: FromRequestParts` 与 axum 0.7（`#[async_trait]` 实现）签名不一致 → 引入 `use axum::async_trait;` 并在 `impl` 上标注 `#[async_trait]`，保持 `async fn from_request_parts(...)`。
- **F4**：`utils/qrcode_util.rs` 的 `AppError` 缺 `From<std::io::Error>` / `From<image::ImageError>`，且 `dark` 原按 `bool` 使用但 `qr[..]` 返回 `&Color` → 在 `error.rs` 增加 `Io` / `Image` 变体（thiserror 自动派生 `From`）；二维码绘制改用 `Color::select(0u8, 255u8)` 直接得到明暗像素值，彻底规避类型比较。

### 2. 修复采购入库回写事务原子性（T06 硬伤）
采购单审批通过（`approval_service::apply_effect` 的 `purchase_order` 分支）以及采购单创建时的自动通过分支（`purchase_service::create_order`）原本把「订单状态变更」与「N 笔库存回写」分散在 1+N 个独立事务，任一 item 失败会留下“订单已通过但库存部分写入”的不一致。

本版改为**单一事务**：
- 在 `repo/inventory_repo.rs` 新增 `receive_purchase_on_conn(conn, ...)`（不自带事务，由调用方管理）。
- 在 `repo/purchase_repo.rs` 新增 `set_order_status_on_conn(conn, ...)` 与 `get_order_on_conn(conn, ...)`（含 `list_order_items_on_conn`）。
- 上述两处业务分支统一用 `conn.transaction()` 包裹 `set_order_status_on_conn` + 循环 `receive_purchase_on_conn`，保证订单状态与库存回写**要么全成、要么全滚**。
- 配合 `approval_service::decide` 已有的「任务非待审批即拒绝」守卫，避免重复审批导致的二次入库。

### 3. 澄清决策 #11 直通契约（仅改文档，不改代码）
可行方案已确认：审批无匹配规则时走“默认直通”= **自动通过、仅发通知 + 写审计 `approval_auto`（module='approval'）、不插入 `approval_tasks` 行**（task_id=None）。
已将 `docs/本地化LIMS_可行方案.md` 决策 #11 的描述修正为该契约（删除原“写 approval_tasks=通过”表述），与现有代码保持一致（采用方案 A：修正决策表述，改动最小）。

### 4. 版本号升至 0.4.22（三处一致）
- `backend/Cargo.toml` 的 `version` → `0.4.22`
- `backend/build_installer.iss` 的 `#define MyAppVersion` → `0.4.22`
- 运行时版本：`api/mod.rs` 的 `/api/version` 返回 `env!("CARGO_PKG_VERSION")`，随 Cargo.toml 自动变为 `0.4.22`
- 三处现已一致。

### 5. 补充：修复一处被掩盖的编译错误
在 `cargo check` 通过库编译后才暴露的二进制编译错误：`src/tray.rs` 使用 `image::ImageReader`，但在 `image 0.24` 下该类型未重导出到 crate 根，且 `Cargo.toml` 仅开启 `png` 特性（关闭默认特性）。改为 `image::io::Reader`（`io` 模块在 image 0.24 下始终可用），实现零 error。此改动为达成“编译闸零 error”所必需，已记入本报告。

### 6. 新增 / 更新双平台打包与 Docker 支持
- **Windows（Inno Setup）**：修正 `backend/build_installer.iss` 的 `Source` 路径，由指向并行副本 `workload-tool-rust\v0.4.21` 改为指向本目录 `v0.4.22/backend`（`target\release\workload-tool.exe`、`static\*`、`icon.ico`），并修正 `SetupIconFile`；**保持 AppId 不变**以确保覆盖安装上个版本。
- **Linux**：新增 `backend/build_linux.sh`（在 Linux / Docker 内执行，构建前端 + 后端 release 二进制）。
- **Docker**：新增 `backend/Dockerfile`（多阶段：rust 编译 backend `--features console` + 前端 `npm run build` 产出 `static` + 运行时暴露 8000 + 健康检查），构建上下文为 `v0.4.22`（`docker build -f backend/Dockerfile .`）；新增 `backend/docker-compose.yml` 便于 `docker compose up`。

## 二、打包与部署（用户可测试）

### Windows
1. 构建后端：`cd backend && cargo build --release`（生成 `backend/target/release/workload-tool.exe`）。
2. 构建前端：`cd frontend && npm install && npm run build`（生成 `backend/static`）。
3. 打包安装程序：用 Inno Setup 6 编译 `backend/build_installer.iss`，生成 `本地化LIMS-0.4.22-setup.exe`（输出目录 `D:\桌面\工作量统计工具项目\installer`）。
4. 安装后运行，浏览器访问 `http://localhost:8000`，首次以 `admin / admin123` 登录并强制改密。

### Linux / Docker
- 原生：`bash backend/build_linux.sh` 后在 `backend/` 运行 `./target/release/workload-tool`。
- 容器：`cd v0.4.22 && docker build -f backend/Dockerfile -t lims-local:v0.4.22 .` 或 `docker compose -f backend/docker-compose.yml up -d`；数据持久化于挂载卷 `./backend/data`（即 `WORKLOAD_DATA_DIR=/app/data`）。

## 三、已知限制 / 说明

## 五、-B 迭代（修复入库误报"网络错误"，版本升至 0.4.22-B）

本版聚焦一处用户反馈的误导性提示修复，并契合"贴合原工作量统计工具风格、为后续内嵌子模块做准备"的约束（仅改前端错误提示 + 版本号相关文件，不动后端业务逻辑，版本隔离）。

### 1. 问题
本地化LIMS 为后端 Axum 本地 8000 端口 + 同源托管前端的单进程程序，**本地环境无远程网络调用**。但点「入库新批次物料」时前端弹出"网络错误"。经只读核查：后端所有错误均经 `AppError` 规范返回 `(200, {code,message,data:null})`，两条入库路径（库存新增批次 / 采购到货）无 panic、无字段类型不匹配、路由齐全；该"网络错误"实为前端 `client.ts` 错误拦截器的兜底文案——只要后端响应体缺 `message` 字段（多为请求未真正打到本地服务），一律兜底显示"网络错误"，属严重误标。

### 2. 修复（仅改前端错误提示，贴合原工具提示风格）
- `frontend/src/api/client.ts` 响应拦截器：移除裸"网络错误"兜底，改为精准提示：
  - 无响应（连接被拒 / 服务未启动）→ `无法连接本地服务，请确认程序是否正在运行`；
  - 响应体含 `message` → 原样透出业务错误；
  - 有响应但无 `message`（如 400/500 非标准体）→ `请求失败（HTTP {status}）`；
  - 其余兜底 → `请求未到达服务，请稍后重试`。
- 后端 handler / `error.rs` 既有逻辑**未改动**（只读调查已证明正确，保持最小改动、版本隔离）。

### 3. 版本号升至 0.4.22-B（三处一致）
- `backend/Cargo.toml` 的 `version` → `0.4.22-B`
- `backend/build_installer.iss` 的 `#define MyAppVersion` → `0.4.22-B`，`OutputBaseFilename` → `本地化LIMS-0.4.22-B-setup`
- 运行时 `/api/version` 随 `env!("CARGO_PKG_VERSION")` 自动变为 `0.4.22-B`
- **AppId 保持独立值 `A69C2C8C-1D53-4DD3-B737-1DF7B9CF3638` 不变** → 本版安装包覆盖上一版 LIMS 安装，仍不影响原「工作量统计工具」。

- 后端：`cargo build --release`（预期零 error），产物 `backend/target/release/workload-tool.exe`。
- 前端：重新 `npm run build`，产物刷新至 `backend/static/`。
- 安装包：用 `D:\APP\Inno Setup 6\ISCC.exe` 编译 `backend/build_installer.iss`，产物 `installer\本地化LIMS-0.4.22-B-setup.exe`。
