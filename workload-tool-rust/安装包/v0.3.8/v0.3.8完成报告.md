## 🎯 核心成果

### 1. 纯名称导入模式

**设计理念**: 导入后的数据仅存储名称，不自动建立关联

| 实体 | group_id | 关联方式 | 说明 |
|------|----------|---------|------|
| 实验室分组 | 独立ID | 无 | 仅名称列表 |
| 研发项目 | **0** | 手动关联 | 不挂靠任何分组（伪分组或真分组）|
| 方法 | 无 | 类型标签 | 仅关联 method_types |

**优点**:
- ✅ 数据模型清晰：`group_id = 0` 表示未关联
- ✅ 避免伪分组污染：不再创建"研发项目"伪分组
- ✅ 用户可控：导入后用户手动选择关联的实验室
- ✅ 灵活性高：一个项目可关联多个实验室（M:N）

## ⚠️ 不兼容变更

### 导入后数据结构变化

**v0.3.6/v0.3.7**:
```sql
-- 导入后项目挂靠伪分组
INSERT INTO project_groups (name) VALUES ('研发项目');
INSERT INTO projects (group_id, name, ...) VALUES (1, 'E003-LC-01', ...);
```

**v0.3.8**:
```sql
-- 导入后项目 group_id=0，不关联
INSERT INTO projects (group_id, name, ...) VALUES (0, 'E003-LC-01', ...);
-- 不再创建"研发项目"伪分组
```

### 前端影响

1. **实验室筛选**: 导入的项目不显示（`group_id = 0`）
2. **手动关联**: 用户编辑项目 → 勾选实验室 → 写入 `project_lab_links`
3. **项目列表**: `GET /api/projects` 默认返回所有项目（包括 `group_id = 0`）

## 📝 重要说明

### 1. 为什么移除自动关联？
- **v0.3.6** 的自动关联逻辑有严重缺陷（仅关联第一个实验室）
- **v0.3.7** 删除了关联但仍挂靠伪分组（数据模型不清晰）
- **v0.3.8** 彻底简化：纯名称存储，用户手动关联

### 2. 手动关联的优势
- ✅ 用户完全可控
- ✅ 支持 M:N 关系（一个项目关联多个实验室）
- ✅ 避免自动关联的错误猜测
- ✅ 数据模型清晰（`group_id = 0` 表示未关联）

### 3. 数据迁移建议
如果从 v0.3.6/v0.3.7 升级：
```sql
-- 将所有挂靠伪分组的项目改为未关联
UPDATE projects SET group_id = 0 WHERE group_id IN (
    SELECT id FROM project_groups WHERE name = '研发项目'
);

-- 可选：删除伪分组（如果没有其他数据依赖）
DELETE FROM project_groups WHERE name = '研发项目';
```
