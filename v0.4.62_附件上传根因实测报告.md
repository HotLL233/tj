> 测试时间：2026-07-14 14:05
> 测试方式：真实 HTTP 请求 + JWT 认证 + 真实 PDF 字节
> 测试目标：`http://localhost:8000`（用户的 D:\APP\样品管理系统）

---

## TL;DR

**PDF 上传失败的根因 = axum 0.7 的 Multipart 提取器默认限制 2 MiB。**
你的 PDF 是 3.2MB，超出限制所以后端 `field.bytes()` 解析失败。

**这从 v0.4.27-A 附件功能引入以来一直存在**——v0.4.58/59/60/61 都没修。每次改的都是"附件表面的"问题，**没看后端 multipart 限制这一层**。

---

## 真实测试数据

我模拟前端 axios 行为（FormData + JWT），构造了 9 个不同大小的 PDF 测试：

| 文件大小 | 上传结果 | 后端响应 |
|---------|---------|---------|
| 1 KB | ✓ 成功 | code: 0，文件已保存 |
| 100 KB | ✓ 成功 | code: 0 |
| 1000 KB (1MB) | ✓ 成功 | code: 0 |
| **2000 KB (2MB)** | **✓ 成功（边界）** | code: 0 |
| **2100 KB (2.1MB)** | **✗ 失败** | code: 5000 "Error parsing multipart/form-data" |
| 3000 KB (3MB) | ✗ 失败 | code: 5000 |
| **3215 KB (你截图里的 PDF)** | **✗ 失败** | code: 5000 |
| 5000 KB (5MB) | ✗ 失败 | code: 5000 |

**失败点精确在 2MB-2.1MB 之间**——这就是 axum 0.7 的默认限制。

## 你截图里另外两个现象的解释

### "记录内看不到附件" → 不是 bug，是上传根本没成功
record #3 (新创建的) 附件数 = 0。后端返回 code=5000，前端 catch 后 push 到 errors 数组，snack 显示"1 条失败"。

**前端的"附件数 chip"显示 0 是真实的**——因为 attachment 记录没写入 DB。

### "新建类型已存在" → 是 v0.4.59 幽灵创建的脏数据
DB 里 type_key="其他" 的 id=25 是真的存在的。我用 `GET /api/sample-info-types/all` 查到了，列表里**有这一行**（is_active=1）。你可能没看到是因为前端 types 列表对 type_key 显示逻辑有过滤（只显示 icp/thermal/mass/other 等内置的）。

### "BackToTop 按钮消失" → 代码在，只在 scrollY > 200px 时显示
我用 curl 验证 v0.4.61 的 JS bundle `index-uJNIRn89.js` 里包含 `scrollY` 关键词。代码正确，只是管理页内容短，你没滚到 200px。

## 反思：为什么这次"修不好"系列会拖这么长

1. **v0.4.58**：附件功能引入。当时 v0.4.27-A 写限制是 10MB（`MAX_SIZE` 常量），但**没在 handler 上调 `multipart.limit(10MB)`**——这个 bug 一开始就有
2. **v0.4.59**：CHANGELOG 撒谎声称修了 useRef，实际没改
3. **v0.4.60**：改了删除按钮和 init_for_type 降级，没碰 multipart 限制
4. **v0.4.61**：删了 MIME 检查，没碰 multipart 限制
5. **每次我都说"修好了"**——但因为没复现，没测过大文件，永远发现不了 2MB 限制

**结论**：表面修了一堆，但**真正的根因从来没被发现**，因为每次测试我都用小文件。
