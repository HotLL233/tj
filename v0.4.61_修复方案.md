## Bug #3（最重要）：新建类型按钮无反应

**严重性**：P0（功能阻断，用户无法创建新检测类型）

### 根因

**文件**：`frontend/src/pages/ManagePage.tsx` 第 1227 行

**当前条件**：
```tsx
{siTypeEdit !== null || siTypeForm.type_key !== '' || siTypeForm.label !== '' ? (
```

**点击「新建类型」按钮时执行的操作**：
```tsx
onClick={() => {
    setSiTypeEdit(null);                                          // siTypeEdit = null
    setSiTypeForm({ type_key: '', label: '', description: '',     // type_key = ''
                    color: '#2e7d32', sort_order: ..., is_active: 1 });
}}
```

**条件计算结果**：
- `siTypeEdit !== null` → `null !== null` → **false**
- `siTypeForm.type_key !== ''` → `'' !== ''` → **false**
- `siTypeForm.label !== ''` → `'' !== ''` → **false**

**结果**：`false || false || false` = **false** → 整个表单 block 渲染 `null`（隐藏）

**为什么之前一直没发现**：
- 编辑类型时 `editSiType` 设置 `siTypeEdit = (type object)` → `siTypeEdit !== null` = true → 表单正常显示
- "新建类型"分支从 v0.4.27-A 引入开始就是这个条件 —— **从那天起「新建类型」按钮就从来没有显示过空表单**！用户看到的"无反应"就是表单完全没渲染

### 修复方案

**方案：增加独立布尔状态 `siTypeNewOpen` 追踪新建表单是否打开**

```diff
+const [siTypeNewOpen, setSiTypeNewOpen] = useState(false);
```

**第 1222 行按钮 onClick**：
```diff
-onClick={() => { setSiTypeEdit(null); setSiTypeForm({...}); }}
+onClick={() => { setSiTypeEdit(null); setSiTypeNewOpen(true); setSiTypeForm({...}); }}
```

**第 1227 行条件**：
```diff
-{siTypeEdit !== null || siTypeForm.type_key !== '' || siTypeForm.label !== '' ? (
+{siTypeEdit !== null || siTypeNewOpen ? (
```

**取消按钮**（第 1236 行）：
```diff
-onClick={() => { setSiTypeEdit(null); setSiTypeForm({...}); }}
+onClick={() => { setSiTypeEdit(null); setSiTypeNewOpen(false); setSiTypeForm({...}); }}
```

**保存成功后**（第 397 行）：
```diff
-if (r.code === 0) { sm('创建成功'); setSiTypeForm({...}); loadSiTypes(); }
+if (r.code === 0) { sm('创建成功'); setSiTypeNewOpen(false); setSiTypeForm({...}); loadSiTypes(); }
```

## Bug #2：提交后附件残留

**严重性**：P1（用户数据错误，旧文件被重复发送）

### 根因

**文件**：`frontend/src/pages/SampleInfoEntry.tsx` 第 323-327 行

`doSubmit` 成功后重置了表单行、时间、页码，但 **没有清空 `pendingFilesRef.current`**。

```tsx
if (submitted > 0) {
    setSnack({ open: true, msg: `成功登记 ${submitted} 条` ... });
    setRows([emptyRow(columns, user)]);
    setSubmittedAt(new Date().toISOString().slice(0, 16));
    setPage(0); load();
    // ← 没有 pendingFilesRef.current.clear() ！
}
```

**后果**：下次用户选新文件时：
```
pendingFilesRef.current.get(0) = [上次的旧文件1, 上次的旧文件2, 本次的新文件]
```
→ 提交时旧文件被再次上传 → 用户认为"上一批附件还在"

### 修复方案

在第 327 行后加一句：
```tsx
setPage(0); load();
pendingFilesRef.current = new Map();  // v0.4.61: 清空待上传文件暂存区
```

### 改动文件

`frontend/src/pages/SampleInfoEntry.tsx` — 1 行

## Bug #1：PDF 上传错误

**严重性**：P1（功能故障，PDF 用户无法上传）

### 可能根因（未完全复现，需验证）

**文件**：`src/api/sample_info_attachment_handler.rs` 第 83-88 行

当前 MIME 验证逻辑：
```rust
if !allowed_types.contains(&file_type.as_str())  // 不在白名单
    && !file_type.is_empty()                       // 不为空
    && file_type != "application/octet-stream"     // 且不是 octet-stream
{
    return Err(AppError::Validation(...));
}
```

**三种可能**：

1. **浏览器传了白名单外的 MIME**（如 `text/plain`、`application/text`）→ 进入 if → 拒绝
2. **后端已修正但用户用的是旧 build**（v0.4.59 白名单没有 octet-stream/x-pdf）
3. **axum Multipart 解析时 Content-Type 丢失** → `field.content_type()` 返回 None → `unwrap_or("application/octet-stream")` → 但第 85 行的 `ft != "application/octet-stream"` 为 false → 不进入 if → 不应拒绝。所以第 3 种可能不成立。

### 修复方案

**方案：完全去掉 MIME 检查，仅保留扩展名验证**

MIME 类型不可靠（大量浏览器/系统差异），扩展名才是稳定的唯一标准。

```diff
-// v0.4.59: 用文件扩展名 + 常见 MIME 类型验证（更可靠）
-let allowed_exts = ["pdf", "doc", "docx"];
-let allowed_types = [
-    "application/pdf",
-    "application/msword",
-    "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
-    "application/octet-stream",
-    "application/x-pdf",
-];
+// v0.4.61: 只检查文件扩展名（MIME 类型不可靠，浏览器/系统差异大）
+let allowed_exts = ["pdf", "doc", "docx"];

    // ...扩展名验证和读取逻辑
-            // MIME 类型宽松检查（避免因浏览器/系统差异拒绝合法文件）
-            if !allowed_types.contains(&file_type.as_str())
-                && !file_type.is_empty()
-                && file_type != "application/octet-stream"
-            {
-                return Err(AppError::Validation(format!("不支持的 MIME 类型: {}", file_type)));
-            }
+
+            // v0.4.61: MIME 类型不可靠，只使用扩展名验证
```

### 改动文件

`src/api/sample_info_attachment_handler.rs` — 删除约 10 行，保留扩展名验证

### 为什么 v0.4.60 没修好

| 问题 | 原因 |
|------|------|
| Bug #3 没被修 | **根本就没发现**。v0.4.60 的修复只改了后端的 init_for_type（降级失败为非致命），但前端 UI 条件 BUG 从未进入分析范围 |
| Bug #2 没被修 | 同样**没有被分析到**。v0.4.60 专注于"删除按钮写错位置"，但没检查 doSubmit 成功后的清空逻辑 |
| Bug #1 声称"修了" | v0.4.59 的 CHANGELOG 说修了（实际只改了 MIME 白名单）。v0.4.60 继承了同样的白名单逻辑，MIME 仍然可能被拒绝 |

**教训**：
1. 只修了用户报告问题的**表面现象**（"上传只显示一个"），没修**深层的代码错误**（表单条件/fref 清空）
2. 没有系统性排查：三个 bug 在同一页面/同一功能模块，应该做全路径检查而不是就事论事修单点
3. CHANGELOG 失真使得用户信任崩溃：用户测试发现 bug 还在，但 CHANGELOG 说"已修"
