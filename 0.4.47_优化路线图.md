# workload-tool v0.4.47 优化路线图

> 编制日期：2026-07-13 | 基于代码审查报告（0.4.47_代码审查报告.md）制定

## 总体策略：渐进式重构，不推倒重来

当前架构骨架（Rust + Axum 三层分层、SQLite + WAL、React + MUI + Vite）是健康的，不需要换。问题出在"肌肉"而非"骨骼"——鉴权散落、组件臃肿、模板重复、测试缺失。按三个阶段推进，投入产出比远高于换架构。

---

## 阶段一：止血（1-2 周，零风险，必须做）

目标：消除所有 P0 安全漏洞，建立统一鉴权防线。

### 1.1 JWT_SECRET 环境变量化（P0）

**当前状态**：
```rust
// src/service/auth_service.rs:17
const JWT_SECRET: &str = "workload-tool-jwt-secret-v0.4.29";
```

**改动**：
1. `config.rs` 中新增 `jwt_secret: String` 配置项，从环境变量 `JWT_SECRET` 或 `config.toml` 读取
2. `auth_service.rs` 中删除硬编码常量，改为从 `AppConfig` 读取
3. 生成脚本或首次启动引导生成随机密钥

**影响范围**：`config.rs`、`auth_service.rs`、`main.rs`（State 注入）
**预估工时**：0.5 天

### 1.2 修复 SQL 注入漏洞（P0）

**当前状态**：
```rust
// src/repo/audit_repo.rs:61-62
Some(m) => ("SELECT COUNT(*) FROM audit_log WHERE module=?1",
    format!("WHERE module='{}'", m.replace('\'', "''"))),
```

**改动**：
1. 移除字符串拼接，统一使用参数化绑定
2. 在入库层对 `module` 字段做白名单校验（仅允许 `"work"`, `"rd"`, `"shared"`）

**影响范围**：`audit_repo.rs`（1 个函数）
**预估工时**：0.5 天

### 1.3 强制首次启动修改默认密码（P0）

**当前状态**：
```rust
// src/config.rs:17
fn default_admin_pass() -> String { "admin123".to_string() }
```

**改动**：
1. 种子数据阶段检测密码是否为默认值
2. 若是默认值，首次启动时生成随机密码并打印到控制台，或弹出修改引导
3. 或者改为环境变量 `ADMIN_PASSWORD` 注入初始密码

**影响范围**：`config.rs`、`seed.rs`、`auth_handler.rs`
**预估工时**：0.5 天

### 1.4 实现全局 Axum 鉴权中间件（P1）

**当前状态**：35 个 handler 中散落着重复的 `extract_claims_from_headers` / `extract_admin_from_headers` 鉴权代码，部分 API（如 `GET /api/settings`）无任何认证保护。

**改动**：
1. 新建 `middleware/auth.rs`，实现 Axum middleware 统一拦截 `/api/*`
2. 白名单路由：`/api/auth/login`、`/api/public/*`
3. 从 Authorization header 提取并验证 JWT，注入 `Claims` 到 Request Extension
4. 逐模块删除各 handler 中的重复鉴权代码，改为从 Extension 获取 Claims
5. 新增 `RequireAdmin` middleware 或 permission guard 函数

**影响范围**：新建 1 个 middleware 文件，修改 `main.rs`（注册 middleware），修改 35 个 handler（删除重复鉴权代码）
**预估工时**：2-3 天

### 1.5 清理前端嵌套目录（P1）

**改动**：
```bash
rm -rf frontend/src/src/
```
**预估工时**：10 秒

## 阶段二：还债（2-4 周，中风险）

目标：解决代码规模和结构问题，提升可维护性。

### 2.1 拆分 ManagePage.tsx（158KB）（P1）

**当前状态**：单个文件承载项目管理、方法管理、分组管理、用户管理、角色管理、系统设置等多个独立功能。

**拆分方案**：

```
frontend/src/pages/
├── manage/
│   ├── ProjectPage.tsx        # 项目管理
│   ├── MethodPage.tsx         # 方法管理
│   ├── GroupPage.tsx          # 分组管理
│   └── index.ts               # 管理模块路由聚合
├── user/
│   ├── UserListPage.tsx       # 用户列表
│   ├── RoleManagePage.tsx     # 角色管理
│   └── index.ts
├── settings/
│   └── SettingsPage.tsx       # 系统设置
└── ...
```

**关键原则**：
- 提取公共表格组件（带排序/筛选/分页的通用 DataTable）
- 提取公共表单组件（带验证的通用 FormDialog）
- 每个子页面独立路由，独立加载

**预估工时**：1-2 周

### 2.2 抽象统计页公共组件（P1）

**当前状态**：StatsPage（78KB）和 SampleStatsPage（74KB）结构高度相似但未复用。

**改动**：
1. 提取公共图表组件：`ChartCard`、`DateRangeFilter`、`SummaryCard`、`PieChartPanel`、`BarChartPanel`
2. 两个页面通过配置驱动，传入不同的数据源和图表定义
3. 通用数据查询 Hook：`useStatsData(config)` 封装 API 调用、加载态、错误处理

**预估工时**：1 周

### 2.3 统一错误响应格式（P2）

**当前状态**：`backup_handler.rs` 等多处使用 `ApiResponse::ok_msg(format!("{}", e))` 返回业务错误，HTTP 200 + code=0，前端无法区分成功/失败。

**改动**：
1. 全面禁止用 `ok_msg` 返回错误
2. 错误统一走 `AppError` → `IntoResponse`，返回正确的 HTTP 状态码
3. 前端 `client.ts` 统一错误拦截器，按状态码分类处理（401 跳登录、403 提示无权限、500 提示系统错误）

**预估工时**：2-3 天

### 2.4 前端引入状态管理库（P3）

**当前状态**：前端仅靠 `UserContext` + `localStorage` 管理状态。

**方案**：引入 Zustand（轻量、TypeScript 友好、无 boilerplate）

```
frontend/src/stores/
├── authStore.ts      # 用户信息、token、登录态
├── settingsStore.ts  # 系统设置缓存
└── ...
```

**预估工时**：2-3 天

### 2.5 清理 .bak 历史文件（P2）

```bash
find frontend/ -name "*.bak" -delete
```

## 阶段三：升维（1-2 月，长期投入）

目标：消除技术债务，建立持续质量保障体系。

### 3.1 Excel 导出模块策略模式重构（P2）

**当前状态**：`export_data.rs`（706 行）+ `export_write.rs`（974 行），11 个 Sheet 的写入逻辑大量重复模板代码。

**方案**：

```rust
// 定义 Sheet 抽象
trait SheetWriter {
    fn sheet_name(&self) -> &str;
    fn headers(&self) -> &[&str];
    fn column_widths(&self) -> &[f64];
    fn write_data(&self, worksheet: &mut Worksheet, data: &SheetData) -> Result<()>;
}

// 每个 Sheet 一个实现
struct SummarySheet;
struct WorkRecordSheet;
// ...

// 统一导出引擎
struct ExportEngine {
    sheets: Vec<Box<dyn SheetWriter>>,
}
```

**预估工时**：1 周

### 3.2 Service 层加厚（P2）

**当前状态**：service 层只有 4 个文件，大量业务逻辑直接写在 handler 中。

**改动**：
- 将 `record_handler.rs` 中的业务校验逻辑下沉到 `record_service.rs`
- 将 `sample_info_handler.rs` 中的数据处理逻辑下沉到新 `sample_info_service.rs`
- 将 `stats_handler.rs` 中的聚合计算逻辑下沉到 `stats_service.rs`
- Handler 回归薄层：**解析请求 → 调 service → 返回响应**

**预估工时**：1-2 周
